Token Conventions

Best practices

모든 세부사항을 정의할 필요는 없습니다. 예를 들면, border width는 border 스코프에서 디펜던시가 높고 변경이 어렵지 않기 때문에 약속을 하지 않는 편이 좋습니다. 볼륨이 커질 때 약속을 해도 충분하다. 문제 해결을 위한 도구의 복잡도를 관리하세요.

Namespace

라이브러리와 충돌을 방지하기 위해 사용합니다. 프로덕션이 아니기 때문에 여기에서는 namespace를 생략합니다.

Component Tokens

컴포넌트 토큰은 상대적으로 변경 가능성이 높고 범용성이 낮기 때문에 컴포넌트 스코프에서 정의합니다.

About exception cases

1~2번 사용하는 예외케이스를 토큰으로 정의하지 않습니다. 컴포넌트 스코프에서 보충합니다.

Naming guideline

--[namespace]-[component / category]-[attribute]-[status]

값을 직접적으로 이름에 포함하지 마세요. 추상화를 통한 장점의 유실.

중요도 Primary Secondary Tertiary
강도 weak moderate strong extreme
instant / immediate quickly slowly deliberate
system status error success warning info updated (new) loading / fetching ?

    Layouts - Width

    페이지 레벨과 섹션 레벨의 레이아웃.

    유연한 그리드 시스템

    모바일 퍼스트(가로 스크롤 트랜드) 기조로m 12 column grid 등 엄격한 그리드 시스템을 정의하지 마세요. 12 칼럼 그리드는 온라인 프로덕트에 어울리지 않으며 모바일 퍼스트 시대에도 어울리지 않습니다. 목록은 언제나 가로 스크롤로 변경하기 쉬워야 합니다. 2개, 3개, 4개의 칼럼을 위한 칼럼 시스템을 설계하세요.

    페이지 모듈을 통한 맥락 구분

    맥락의 전환(페이지의 역할의 구분??)을 표현하기 위해 레이아웃 모듈 또는 배경색의 대비를 활용할 수 있습니다. 1. 컨테이너 너비(비밀번호찾기와 로그인/회원가입) 2. 배경색(설정, 글쓰기, 소개 등)

    가이드라인?

    • breakpoint small에서는 layout-width가 %(퍼센티지)로 변경됩니다.
    • 섹션, 아티클 그리고 모달, 드롭다운 등의 컴포넌트도 작은 화면에서는 페이지 맥락이 될 수 있음을 고려합니다.
    • wide / narrow, card layout, form layout
    • small-scale layout (polaris 참고) 아바타 유무
