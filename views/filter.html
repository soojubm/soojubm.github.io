프레젠테이션

presentation 컴포넌트가 필터의 상태나 적용 방법에 많은 영향을 준다. 필터 옵션의 presentation 컴포넌트는 환경에 최적화된 경험을 위해 변경하기 쉬워야 한다.

  • Dropdown with chip group: desktop에서 상. moblie에서 중. desktop과 mobile에 동일한 경험과 낮은 개발 비용. 막연한 탐색에 좋다.
  • Sheet: desktop에서 하. mobile에서 상. ???? 왜 넣었었지.
  • Modal: desktop과 mobile에서 중. desktop과 mobile에 동일한 경험과 낮은 개발 비용. 상세한 탐색.

필터의 적용

  • Instant updates. 각각의 필터를 즉시 적용 (onChange) - 더 좋은 사용자 경험 but 서버 요청, 데이터
  • Batch updates. 적용 버튼을 클릭할 때 적용 (onSubmit) - 주로 multiple filters. single filter에서도 사용하는 케이스도 있지만.

적용된 필터의 상태

  • Multiple filters: 필터 트리거 버튼에 적용된 필터의 갯수 표시. 전환에 필수불가결한 옵션이 존재하는 경우(다소막연한데 세부적인 검색ㅇ ㅣ필요할 때.).
  • Single filter: 액티브 스타일과 레이블을 변경. (todo multiple)
  • 적용된 필터 상태를 removable chips로 추가 제공(case. 오늘의집, 네이버카페) => 과연 효용이 있는 인터페이스인가?

궁굼이 & 생각

  • 사용자는 적용된 필터 옵션을 초기화하기 위해 새로고침을 할까?
  • multiple filters에서 각각 목록은 설정(setting)에서 사용하는 패턴과 같음.
  • 하나의 화면에서 모든 옵션 목록을 나열할 때 옵션 별로 다른 컴포넌트로 제공하여 옵션 하나 하나에 주의를 집중중시킬 수 있다. dropdown, chip, radio 등의 혼합 => 디자인과 개발 비용이 증가하는 만큼 사용자 경험이 월등한가? (case. airbnb) 아니면 다른 의도?
  • 필터를 적용하고 초기화하는 버튼 그룹의 정렬은 dialog에서 버튼 그룹과 달라질 수 있다. hero 컴포넌트와는 동일하다.
  • range UI는 얼마나 사용자 친화적인가
  • 모든 필터 옵션을 세로 목록형 가로 스크롤로 나열할 수도 있다. 큰 화면에서 유용하다. (case. bilibili)
