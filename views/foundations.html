Trustworthy Visual cue

친숙하고 신뢰할 수 있는 서비스 경험을 위해 예측 가능하고 일관되게 기표를 사용합니다. UX 부채를 최소화하세요.

우리가 해결하려는 문제는 사용자 인터페이스가 아니다. 진짜 문제(또는 콘텐츠)에 집중할 수 있는 시스템.

  • 기술과 환경의 한계를 인정합니다. 디바이스나 브라우저의 크기에 종속되지 않는 패턴 설계. 미래의 사용자는 자동차, 냉장고에서 (도메인 특수성이 있지만). 모든 환경이 고려된, 모든 환경이 고려되었다는 것은 작은 화면을 기준한다는 것.
  • 토큰 변경 만으로 다양한 테마에 대응/li>
  • 변경 사항 기록과 누적 (새로운 동료의 온보딩과 제품 개선 히스토리)
  • 사용자(동료) 피드백 (디자인 시스템에 기여/크래딧을 통한 동기부여)

Layout

사용자의 화면 크기를 한정하지 마세요.

새로운 플랫폼이나 새로운 디바이스가 출시되더라도 쉽고 빠르게 대응할 수 있어야 합니다. 특정 크기가 아니라 환경에 구애받지 않는 경험과 인터페이스를 설계한다는 태도가 필요합니다.

pain: 왜 많은 디자이너들은 시안의 사이즈를 스스로 정의하지 못 하는가? => 375px 등의 고정된 가로 크기는 전혀 중요한 것이 아님.. 편의상 정의하는 하나의 화면 크기, 즉 대시보드의 크기를 관점으로 접근하는 것이 아니라(이렇게 되면 실제 사용자의 문제를 해결하거나 사용자의 화면을 아름답고 편리하게 만드는 게 아닌 디자이너 자신의 디자인 툴을 사용하기 쉽게 만드는 일에 매몰). 자동으로 말줄임표를 만든다거나. 이것이 무슨 문제를 해결하는가? 말줄임표가 발생하면 레이블의 의미를 전달하지 못 하기 때문에 또 다른 솔루션의 필요해짐. 엄격한 ux writing 가이드 만드는 일이 유아이 툴의 자동화보다 중요한 일이 되어야 함. +

  • 데스크탑 환경에서나 유효한 hover 형태를 디자인하는 데에 많은 시간을 들일 필요가 없습니다. 시각 이상자를 위한 형태. 크기, 깊이, 테두리의 제한된 용법은 제품 개발 속도를 높인다.

그리드 시스템.

12칼럼 그리드를 쓰는 이유가 무엇. 장점이 없음. 2~4개의 칼럼을 가지는 레이아웃 케이스와 간격 시스템을 정의하고 가로 스크롤을 활용.

디자인 이론에 근거한 그리드 시스템, 예를 들면 12칼럼 그리드는 코드 환경에서 사용자의 다종다양한 화면 크기에 대응하기 적합하지 않다. 그리드 시스템을 다양한 화면 크기에 적용하기 위해서는 고려해야 하는 경우의 수가 많다. 그리고 이 시스템은 디자이너에게만 유효하게 동작한다.
