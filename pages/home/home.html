서버로의 네트워크 왕복 시간 (전파 지연 시간)은 100ms입니다. 서버 응답 시간은 HTML 문서의 경우 100ms, 기타 모든 파일의 경우 10ms입니다. HTML 콘텐츠를 사용할 수 있게 되면 브라우저가 바이트를 파싱하여 토큰으로 변환한 후 DOM 트리를 빌드합니다. 이미지는 페이지의 초기 렌더링을 차단하지 않지만 가능한 한 빨리 이미지를 그려야 합니다. onload 이벤트는 페이지에 필요한 모든 리소스가 다운로드되고 처리된 지점을 표시합니다. 이 시점에서 로딩 스피너가 브라우저에서 회전을 멈출 수 있습니다 (워터폴의 빨간색 세로선). 일반 HTML 예와 달리 CSSOM을 생성하기 위해 CSS 파일을 가져오고 파싱해야 하며, 렌더링 트리를 빌드하기 위해 DOM 및 CSSOM이 모두 필요합니다. 페이지에는 파서 차단 JavaScript 파일도 포함되어 있으므로 CSS 파일이 다운로드되고 파싱될 때까지 domContentLoaded 이벤트가 차단됩니다. JavaScript가 CSSOM을 쿼리할 수 있으므로 JavaScript를 실행하려면 다운로드될 때까지 CSS 파일을 차단해야 합니다. 브라우저가 스크립트 태그에 도달하는 즉시 차단하고 CSSOM이 생성될 때까지 대기하기 때문입니다. 첫째, 모든 인라인 스크립트는 파서를 차단하지만 외부 스크립트의 경우 'async' 키워드를 추가하여 파서 차단을 해제할 수 있습니다. 인라인 처리를 취소하고 시도해 보겠습니다. domContentLoaded 시간은 이전 예와 사실상 동일합니다. JavaScript를 비동기로 표시하는 대신 CSS와 JS를 모두 페이지 자체에 인라인 처리했습니다. 이로 인해 HTML 페이지가 훨씬 커지지만 장점은 페이지가 외부 리소스를 가져오기 위해 기다릴 필요가 없다는 것입니다. 페이지에 모든 것이 바로 있습니다. CSS와 자바스크립트를 병렬로 전송할 수 있기 때문입니다. 주요 렌더링 경로의 특성을 파악하면 주요 리소스를 식별하고 브라우저가 이에 대한 가져오기를 예약하는 방식을 이해할 수 있습니다. 예시를 계속 살펴보겠습니다. 사이트 개발자와 채팅한 결과, 페이지에 포함한 JavaScript를 차단할 필요가 없다는 것을 알게 되었습니다. 페이지 렌더링을 차단할 필요가 없는 몇 가지 분석 및 기타 코드가 있습니다. 이렇게 하면 'async' 속성을 스크립트 태그에 추가하여 파서 차단을 해제할 수 있습니다. 비동기 스크립트에는 다음과 같은 여러 장점이 있습니다. 스크립트가 더 이상 파서를 차단하지 않으며 주요 렌더링 경로의 일부가 아닙니다. 다른 중요한 스크립트가 없으므로 CSS에서 domContentLoaded 이벤트를 차단할 필요가 없습니다. domContentLoaded 이벤트가 빨리 실행될수록 다른 애플리케이션 로직도 빨리 실행될 수 있습니다.
