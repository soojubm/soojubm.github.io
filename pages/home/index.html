Constraint-driven Design 우리는 제약을 통해 설계한다. 모든 디자인은 명확한 제약 안에서 이루어진다. 제약은 선택을 줄이고, 품질과 일관성을 강제하는 기준이다. 👉 핵심: 좋은 디자인을 선택하는 것이 아니라, 잘못된 디자인을 불가능하게 만든다
Progressive & Scalable 우리는 단순하게 시작하고, 점진적으로 확장한다. 완성된 시스템을 가정하지 않는다. 필요에 따라 구조와 패턴을 확장한다. 👉 핵심: 처음부터 완벽하게 만들지 않는다. 대신 확장 가능하게 만든다
1. Constraint-driven Design 한계를 설정하기
우리는 선택을 줄이기 위해 제약을 만든다. 제약은 속도와 일관성을 만든다.
A11y 접근성은 기본 제약이다. 충족하지 않는 디자인은 허용하지 않는다. 색상에 의존하지 않고 형태, 대비, 구조로 상태를 표현한다.
Mobile First 가장 제약이 큰 환경에서 시작한다. 환경에 종속되지 않는 일관된 경험을 만든다.
2. Universal Experience 환경에 독립적인 경험
우리는 디바이스와 브라우저에 종속되지 않는 경험을 설계한다.
Platform-agnostic 특정 환경에 최적화된 표현보다 모든 환경에서 일관된 구조를 우선한다.
Consistent Patterns 동일한 문제는 동일한 방식으로 해결한다. 패턴의 반복이 일관성을 만든다.
3. Progressive & Scalable 점진적으로 확장하기
우리는 단순하게 시작하고 필요에 따라 확장한다.
Start Small 초기에는 최소한의 구조와 패턴만 정의한다.
Scale by Need 실제 사용과 요구에 따라 컴포넌트와 패턴을 추가한다.

Architecture Layers 추상에서 구체로. 아래로 내려갈수록 도메인에 가까워지고, 재사용 범위는 좁아진다.
레이어 판단 기준 다른 도메인이나 서비스에서도 쓸 수 있는가? → Semantic. 이 서비스에만 존재하는 개념인가? → Concrete. 폴더 구조가 이 경계를 구조적으로 강제한다. semantics/ = 도메인 독립, domains/ = 도메인 종속.
새 요소가 필요할 때 Abstract 없으면 만든다 → 도메인 무관하면 Semantic으로 고정한다 → Concrete에서 조합한다
레이어 정의 예시 / 위치
LAYER 1 Abstract
HTML 요소 수준의 가장 작은 단위. 의도 없이 props로만 변형되는 범용 빌딩 블록.
mm-button, mm-text, mm-tag, mm-icon-button src/components/{name}/
↓ extends — Semantic은 Abstract를 상속해 역할을 고정한다
LAYER 2 Semantic
의미가 있고 도메인에 독립적인 단위. 어떤 서비스에서도 쓸 수 있는 개념. variant 대신 이름이 용도를 표현.
mm-read-more-button, mm-close-button, mm-caption, mm-accent-tag src/components/{name}/semantics/
↓ composes — Concrete는 Abstract·Semantic 컴포넌트를 조합해 완성한다
LAYER 3 Concrete
특정 도메인·서비스에 종속된 조합. 다른 서비스에서 그대로 가져다 쓸 수 없는 것.
mm-comment-input, mm-user-row, mm-prompt-input src/components/domains/
이 방법론의 한계
"다른 서비스에서 쓸 수 있는가"의 판단은 여전히 주관적이다 mm-search-suggestions는 어떤 서비스에나 있는 검색 UI다 — Semantic인가? 아니면 이 서비스의 검색 동작에 종속된다 — Concrete인가? 기준이 명확해도 경계 사례에서는 판단이 갈린다. 폴더 구조가 강제할 수 있는 건 배치 이후다. 배치 전 판단은 사람이 한다.
Concrete가 도메인을 넘어 재사용되면 레이어가 흔들린다 mm-comment-input이 댓글과 DM 두 화면에서 모두 쓰인다면? 처음엔 도메인 특화였다가 점점 범용이 된다. Concrete를 Semantic으로 격상하는 시점과 기준이 없으면 domains/ 안에 사실상 Semantic인 컴포넌트가 쌓이기 시작한다.
Semantic 레이어가 팽창하기 쉽다 "도메인 무관"의 기준이 느슨하면 모든 컴포넌트가 Semantic 후보가 된다. 작은 변형마다 새로운 Semantic 컴포넌트가 추가되면 semantics/ 디렉토리가 비대해지고, Abstract 컴포넌트의 variant로 충분히 해결할 수 있는 것들이 별도 클래스로 분산된다.
